# Internals

Read-only artifacts about the Grinta codebase that are useful when
planning refactors, layering decisions, or decomposition work. Nothing
in this directory is required to run, install, or test Grinta.

## `import-manifest.json`

A static snapshot of the import graph for every Python module under
`backend/` and `launch/`. Generated by
[`scripts/discover_public_imports.py`](../../scripts/discover_public_imports.py),
which is an AST-only, read-only analysis pass — it does **not** import
the project and does **not** execute project code.

The manifest is consumed by the file-size decomposition work
documented in [`../ARCHITECTURE.md`](../ARCHITECTURE.md). Before any
monolith is split, the manifest tells us which files import the
target module and which names they reach for, so the re-export shim
for the old path can be planned with zero guesswork.

## `confirmation-autonomy.md`

The architecture note for the confirmation-state ownership fix. It records why
orchestration is the only layer allowed to set `AWAITING_CONFIRMATION`, and why
runtime execution only reads that state or blocks unsafe actions.

### How to regenerate

```bash
python scripts/discover_public_imports.py
# or
python scripts/discover_public_imports.py --output docs/internals/import-manifest.json
```

The default invocation excludes `backend/tests/` (tests are
*consumers* of the public surface, not providers). Use
`--include-tests` to see the test-import graph as well, or `--quiet`
to suppress the human-readable summary and only emit JSON.

### Schema

Top-level keys (see `schema_version` for the current version):

| Key | Description |
| --- | --- |
| `schema_version` | Integer; bump on any breaking change to the shape. |
| `generated_at` | UTC ISO-8601 timestamp of generation. |
| `repository_root` | Absolute path to the repo root used for the walk. |
| `options` | CLI flags in effect when the manifest was generated. |
| `summary` | Aggregate counts: module count, import count, classification, parse failures. |
| `biggest_files` | Top 25 modules by `byte_size`. |
| `most_imported` | Top 25 modules by external importer count. |
| `public_surface` | For each known monolith, the list of names that other modules import, with import counts. |
| `modules` | Dict keyed by dotted module path; see below. |
| `parse_failures` | Files that could not be parsed (usually empty). |

Each entry in `modules` has:

| Field | Description |
| --- | --- |
| `module_path` | Dotted module path, e.g. `backend.cli.tui.app`. |
| `file_path` | Repo-relative POSIX path. |
| `byte_size` | File size in bytes. |
| `loc` | Physical line count. |
| `class_count` | Top-level `class` definitions. |
| `def_count` | Top-level `def` / `async def` definitions. |
| `importer_count` | Length of `imported_by`. |
| `intra_package_imports` | Deduplicated internal modules this file imports (not including itself). |
| `imported_by` | List of `{importer, lineno, names, is_type_checking}` records. |

### Caveats

* **Static, not dynamic.** Anything reached through `importlib.import_module(...)`,
  `runpy.run_path(...)`, or `__getattr__` aliases will not appear.
  For Grinta, that is limited to the `launch/entry.py` shim which uses
  `runpy.run_path` to invoke `backend/cli/entry.py`.
* **TYPE_CHECKING imports are flagged, not removed.** The manifest
  preserves them with `is_type_checking: true` so the decomposer can
  decide whether a runtime shim is required.
* **No imports are executed.** The script only parses ASTs. It is safe
  to run on a dirty working tree, on a half-installed interpreter,
  or even with the venv unset.

### Current snapshot (last generated)

The committed `import-manifest.json` is regenerated on demand — it is
not auto-updated by CI. If you are about to plan or execute a
decomposition PR, regenerate it first so the importer list reflects
the current `main`.
