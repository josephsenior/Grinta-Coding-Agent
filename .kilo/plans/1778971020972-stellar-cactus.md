# Plan: Fix TUI User-Message Decoration + Improve Turn-End Summary

## Context

Two visual problems exist in `backend/cli/tui/app.py` only (the CLI Rich
renderer in `event_renderer.py` is unaffected — it uses its own `Panel` /
`Group` pipeline):

1. **User-message accent indicator** — every visual line of a multi-line user
   message should begin with a coloured `▌` accent, but only the first line
   currently shows it correctly.
2. **Turn-end summary banner** — after each agentic turn the TUI writes a thin
   divider (`───`) and a summary (`N tools · X.Xs`). The CLI equivalent in
   `event_renderer.py` already exposes per-turn token count, cost, and call
   deltas via `_turn_stats_text()`, but the TUI never surfaces them and the
   present banner is visually flat.

---

## File at Risk

`backend/cli/tui/app.py` — the only file that needs to change.

Existing tests in `backend/cli/tui/tests/test_headless.py` are read-only during
planning; `test_headless.py` line 179 exercises `add_user_message` and
`write_log` via a live-pilot run.

## Proposed Changes

### Change 1 — `add_user_message` (lines 439–461): guarantee the accent
appears on every wrapped line

**Current symptom**

The method splits the user text into lines, wraps each, and then accumulates
the result one line-at-a-time with:

```python
accent = Text('▌', style='bold #91abec')
result = Text()
for i, line in enumerate(wrapped_lines):
    if i > 0:
        result.append('\n')           # ← plain-string newline segment
    result.append(Text.assemble(accent, ' ', line))  # per-line accented segment
result.append('\n')                   # trailing spacer
```

The intent is correct — the accent token is prefixed before each line — but
the interleaving of plain-string `'\n'` segments with `Text.assemble(…)`
segments into a single mutable `Text` object can misalign the accent span when
the object is eventually serialised through Textual's `Static` widget update
path (`Group(*self._history)` → `Static.update(Group(…))`).  In practice this
means multi-line user messages end up with the accent on the first line only.

**Proposed code**

Build each visual line as a self-contained `Text` token first, then join them
into one flat string with newlines.  This guarantees the accent char lands in
the character sequence before every line without depending on Rich's
mutating-Text outlier behaviour:

```python
accent = Text('▌', style='bold #91abec')

# Build per-line segments in a list (each is a fully-formed, accented Text).
line_tokens = [
    Text.assemble(accent, ' ', raw_line)
    for raw_line in wrapped_lines
]

# Join the plain-text representation; accent char is baked into every segment.
result = Text('\n'.join(t.plain for t in line_tokens)) if line_tokens else Text()
result.append('\n')    # trailing spacer after user message — preserved.
```

**Why this works**

The `▌` character becomes a plain character in the resulting string, not a
floating Rich style span that can flatten on re-traversal.  Every wrapped
line literally begins with `▌ `, so no matter how Static or Group render the
object the accent cannot be lost on any line that is >0 width.  The 42-char
reduced-width implementation does not affect this logic.

### Change 2 — Turn-end summary banner (lines 1813–1822): richer metrics with
wider divider

**Current banner**

```
──────────────────────────────────────────
  N tools · X.Xs
```

**Proposed banner**

```
  ══════════════════════════════════════════════════════════════
   N tools  ·  X.Xs  ·  X.XXK tokens  ·  N LLM calls  ·  $X.XXXX
```

Changes required inside the `if self._tools_in_turn > 0` block:

1. Replace the 40-char `'─'` thin rule with a 42-char `'═'` thick rule so the
   end-of-turn boundary is visually heavier than the per-tool activity
   `───` divider.
2. Compute the three per-turn deltas from the snapshots that are already
   captured at turn start (`_turn_start_tokens`, `_turn_start_cost`,
   `_turn_start_calls`) against the current HUD values:

   ```python
   token_delta = self._hud.state.context_tokens - self._turn_start_tokens
   cost_delta  = self._hud.state.cost_usd     - self._turn_start_cost
   call_delta  = self._hud.state.llm_calls    - self._turn_start_calls
   ```

3. Build the summary parts list in priority order — the fields that are always
   significant (tool count, wall-clock) go first, followed by token delta,
   LLM-call delta, and cost delta when they are non-zero:

   ```python
   parts: list[str] = [
       f'{self._tools_in_turn} tool' + ('s' if self._tools_in_turn != 1 else ''),
       f'{elapsed:.1f}s',
   ]
   if token_delta > 0:
       parts.append(f'{self._hud._format_tokens(token_delta)} tokens')
   if cost_delta > 0.0001:
       parts.append(f'${cost_delta:.4f}')
   if call_delta > 0:
       parts.append(f'{call_delta} call' + ('s' if call_delta != 1 else ''))
   ```

4. Render the divider and summary with `dim #32416a` (CLR_TURN_RULE) and
   `dim #969aad` (CLR_META) — both are already defined in `theme.py` and the
   `deep-system-instrumentation` preset, so no new colour constants are needed:

   ```python
   divider_segment = Text('═' * 42, style='dim #32416a')
   summary_segment  = Text('  ' + '  ·  '.join(parts), style='dim #969aad')
   self._tui._write_log(Text.assemble('\n', divider_segment, '\n', summary_segment, '\n'))
   ```

## Design Notes

- `═══` (U+2550 box-drawing double horizontal) is chosen to outweight the
  existing per-tool `───` divider that `format_activity_turn_header` already
  emits; the two visually form a hierarchical pair: thin rule for the tool
  section header, thick rule for the turn-end boundary.
- Accent colour `bold #91abec` on the user-message left bar is the existing
  brand-blue constant already used by `CLR_USER_BORDER`; no new colour is
  introduced.
- `HUDBar._format_tokens` (already used in
  `event_renderer._turn_stats_text()`) is reused for token-formatting
  consistency; it safely clamps `1.2K tokens` rather than showing raw integers.
- Neither `event_renderer._turn_stats_text` nor `hud._format_tokens` are
  modified. The TUI computes the same deltas locally, ensuring the
  implementation path diverges cleanly from the existing event-renderer chain.

## Verification Path

`backend/cli/tui/tests/test_headless.py` already drives `add_user_message`
through a live Textual pilot; the same call must succeed without raising.

For Change 2, any end-to-end orchestrator run that reaches an idle/terminal
state will render the banner. A headless round-trip test that drives the
`_process_event` path end-to-end is the cleanest signal — the TUI already
provides event-drain and pilot-pause breakpoints that let an automated harness
inspect the post-turn banner without an actual TTY terminal.

---

## Non-Scope / Deferred

- The CLI `event_renderer` does not currently emit a per-turn-end summary
  (the `_turn_stats_text` helper is unused in the live path); changing its
  call-site is out of scope for this plan.
- Adding a named constant for the turn-end divider to `layout_tokens.py` or
  `theme.py` is deferred — the 42-char inline string is small enough that a
  `# P2 — extract to constant` TODO is acceptable.
- Reusing `CLIEventRenderer._turn_stats_text` directly in the TUI path is
  deferred: the TUI collect the turn-start snapshots (`_turn_start_tokens`,
  `_turn_start_cost`, `_turn_start_calls`) without touching the event-renderer
  data model, so introducing a shared import would couple the two layers
  prematurely.
